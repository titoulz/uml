Introduction

Cette page présente l'ensemble des diagrammes de séquence de l'application Allo-Halal, une plateforme de réservation de restaurants halal à Dijon et ses environs.

🎯 Objectif de l'application

Permettre aux clients de réserver facilement dans des restaurants halal certifiés, et aux restaurateurs de gérer leurs réservations en temps réel.

🛠️ Technologies utilisées

  • Frontend : HTML5, CSS3, JavaScript (Vanilla)
  • Backend : Node.js, Express.js
  • Base de données : SQLite (better-sqlite3)
  • Sécurité : bcrypt (hashage des mots de passe)
  • Carte interactive : Leaflet.js + OpenStreetMap

🔐 Diagrammes d'Authentification

1. Inscription d'un nouveau utilisateur

Description : Processus complet d'inscription d'un nouveau client ou restaurant.

Acteurs : Utilisateur, Interface Web, AuthService, API Backend, Base de données, SessionManager

Cas d'erreur gérés :

  • Champs invalides ou manquants
  • Email déjà existant dans la base

Résultat : Compte créé + connexion automatique + son de bienvenue + redirection vers l'accueil

2. Connexion

Description : Connexion d'un utilisateur existant avec gestion de la persistance de session.

Options : "Se souvenir de moi" (localStorage) ou session temporaire (sessionStorage)

Cas d'erreur gérés :

  • Champs vides
  • Email non trouvé
  • Mot de passe incorrect

Sécurité : Vérification avec bcrypt.compare()

3. Déconnexion

Description : Déconnexion simple de l'utilisateur.

Action : Suppression de la session (localStorage + sessionStorage) et rechargement de la page

4. Modification du profil

Description : Modification du nom, email et/ou mot de passe de l'utilisateur.

Cas d'erreur gérés :

  • Utilisateur non connecté
  • Utilisateur non trouvé
  • Mot de passe actuel manquant (si changement de mdp)
  • Mot de passe actuel incorrect

Sécurité : Vérification du mot de passe actuel avant de permettre le changement

5. Suppression du compte

Description : Suppression définitive du compte utilisateur.

Sécurité : Confirmation obligatoire avant suppression

Résultat : Compte supprimé + déconnexion automatique + redirection vers l'accueil

📅 Diagrammes de Gestion des Réservations

6. Création d'une réservation

Description : Processus complet de création d'une réservation par un client.

Prérequis : L'utilisateur doit être connecté

Informations requises :

  • Type de service (sur place / à emporter)
  • Date et heure
  • Nombre de personnes
  • Nom, téléphone, email
  • Demandes spéciales (optionnel)

Résultat : Réservation créée avec statut "pending" (en attente de validation par le restaurant)

7. Annulation d'une réservation (Client)

Description : Annulation d'une réservation par le client.

Sécurité : Confirmation demandée avant annulation

Résultat : Statut de la réservation changé en "cancelled" + liste actualisée

8. Acceptation d'une réservation (Restaurant)

Description : Acceptation d'une réservation par le restaurateur.

Contrôles d'accès :

  • Utilisateur doit être connecté
  • Utilisateur doit avoir le rôle "restaurant"
  • Utilisateur doit avoir un restaurantId associé

Résultat : Statut changé en "confirmed" + notification au client

9. Refus d'une réservation (Restaurant)

Description : Refus d'une réservation par le restaurateur.

Sécurité : Confirmation demandée avant refus

Résultat : Statut changé en "refused" + notification au client

📊 Statuts des réservations

🟡 Pending

En attente de validation par le restaurant

🟢 Confirmed

Acceptée par le restaurant

🔴 Refused

Refusée par le restaurant

⚫ Cancelled

Annulée par le client

🔒 Architecture de sécurité

Hashage des mots de passe

  • Utilisation de bcrypt avec un coût de 10
  • Les mots de passe ne sont jamais stockés en clair
  • Comparaison sécurisée avec bcrypt.compare()

Gestion des sessions

  • localStorage : Session persistante (si "Se souvenir de moi")
  • sessionStorage : Session temporaire (fermée à la fermeture du navigateur)
  • Contenu : {userId, userName, userEmail, userRole, restaurantId, isLoggedIn, loginTime}

Contrôles d'accès

  • Pages client : vérification du rôle "client"
  • Pages restaurant : vérification du rôle "restaurant" + restaurantId
  • API : validation des données côté serveur
  • Base de données : contraintes d'intégrité (FOREIGN KEY, CHECK)
